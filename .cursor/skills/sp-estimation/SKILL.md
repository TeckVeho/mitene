# SP見積りスキル

## 基本定義

**1 SP = 1時間** (実装 + レビュー対応を含む)

SPは **1以上の整数** で割り当てる。

---

## 見積り手順

```
SP見積りチェックリスト:
- [ ] 1. クイック判定フローでSPゾーンを把握
- [ ] 2. 6軸でスコアを算出し精密化
- [ ] 3. 整数に丸める
- [ ] 4. セルフチェック3問を確認
- [ ] 5. 根拠を1行でコメント
- [ ] 6. setsp スクリプトで GitHub Projects に反映
```

---

## Step 1: クイック判定フロー

6軸評価の前にこのフローでSPゾーンをざっくり掴む。

```
Q1: 変更ファイル数は？
  1ファイル         → 1~2 SP ゾーン
  2~3ファイル       → 2~3 SP ゾーン
  4ファイル以上     → 5~8 SP ゾーン

Q2: 新規ロジックの実装は必要？
  既存パターンのコピーで済む → SP据え置き
  新規ロジック必要           → +1 SP

Q3: テストの追加・変更は必要？
  不要 or 既存テストの軽微修正 → SP据え置き
  ユニットテスト追加           → +1 SP
  統合/E2Eテスト追加           → +2 SP
```

クイック判定後、不確実性が高い・スコープが曖昧な場合は Step 2 で精密評価する。

---

## Step 2: 6軸見積りフレームワーク

各軸を 1〜3 でスコアリングし、合計をガイドとして使う。

| 軸 | 1 (小) | 2 (中) | 3 (大) |
|----|--------|--------|--------|
| **コード量** | 〜50行 | 50〜200行 | 200行超 |
| **複雑度** | 直線的なロジック | 条件分岐・変換処理 | アルゴリズム・再帰・状態管理 |
| **テスト** | 確認のみ | ユニットテスト追加 | 統合/E2Eテスト必要 |
| **アーキテクチャ影響** | 既存パターンに従う | 新規関数/モジュール | 設計変更・マイグレーション |
| **外部依存** | なし | 内部API呼び出し | 外部API/サードパーティ |
| **不確実性** | 既知の実装 | 調査が多少必要 | 未知の技術・要件が曖昧 |

**合計スコア → SP変換テーブル:**

| 合計 | SP |
|------|----|
| 6 | 1 |
| 7〜8 | 2 |
| 9〜10 | 3 |
| 11〜12 | 4 |
| 13〜14 | 5 |
| 15〜16 | 6 |
| 17〜18 | 7 |

## Step 3: SPスケールと具体例

### 1 SP (〜1時間)
- typo修正・定数変更・ラベル追加
- CSSの微調整・色・余白の変更
- 既存コンポーネントへの小さなプロパティ追加
- 単純なバリデーションルール追加

### 2 SP (〜2時間)
- シンプルなUIコンポーネント (表示のみ)
- 単一フィールドのCRUDエンドポイント
- 設定ファイル変更 + 関連テスト追加
- 既存テストへのケース追加

### 3 SP (〜3時間)
- 標準的なフォームUI + バリデーション
- 中程度の複雑さのAPIエンドポイント + ユニットテスト
- 既存モジュールへの機能追加

### 4 SP (〜4時間)
- 複数の関連エンドポイントをまとめた機能追加
- 既存機能の大幅な改修 + ユニットテスト
- 設計変更を伴わない中規模リファクタリング

### 5 SP (〜5時間)
- 複数コンポーネントにまたがる機能 (状態管理含む)
- 認証・認可ロジック
- データ変換・集計が必要なAPI
- DBマイグレーション + 対応するビジネスロジック

### 6 SP (〜6時間)
- 新規ページ全体 (UI + API統合 + テスト)
- 複雑なビジネスフローの実装
- 既存設計に影響する大きなリファクタリング

### 7 SP以上
- 新規モジュール全体の設計・実装
- 複数サービスにまたがる機能
- スコアに応じてそのままSPとして出力する

## Step 4: セルフチェック

見積り後に以下の3問に答える。1つでも「No」なら見積りを見直すこと。

| # | 問い | 確認観点 |
|---|------|---------|
| 1 | このSPで、途中で止まらずに完了できるか？ | 依存タスクや調査待ちが混在していないか |
| 2 | スコープ外の作業が混ざっていないか？ | 別issueにすべき作業が含まれていないか |
| 3 | 不明点がある場合、調査タスク(1〜2 SP)として切り出したか？ | 不確実性を同一issueで抱えていないか |

## FE / BE 別の見積り特性

### Frontend (FE) でSPが増える要因
- レスポンシブ対応・多言語対応
- 複雑なフォームバリデーション
- リアルタイム更新 (WebSocket, polling)
- アニメーション・UXの細かい作り込み

### Backend (BE) でSPが増える要因
- トランザクション処理・ロールバック
- バッチ処理・非同期ジョブ
- 外部API連携 (エラーハンドリング込み)
- データマイグレーション (既存データの変換含む)

## GitHub Projects への SP反映

`script/setsp.ps` (bash / Ubuntu / macOS / Git Bash) または `script/setsp.ps1` (PowerShell / Windows)。詳細は `CROSS-PLATFORM.md`。

```bash
# bash (macOS/Linux/WSL)
bash .cursor-shared-skills/script/setsp.ps <issue_url> <sp_value>

# 例
bash .cursor-shared-skills/script/setsp.ps https://github.com/org/repo/issues/123 5
```

スクリプトは以下を自動処理する:
- issueが紐づく GitHub Project を検索
- `SP` / `StoryPoint` / `StoryPoints` にマッチするフィールドを検出
- フィールドタイプ (NUMBER / TEXT / SINGLE_SELECT) に合わせてミューテーション実行

> **前提**: `gh` CLI でログイン済み、`jq` インストール済みであること。

## アンチパターン

| アンチパターン | 対策 |
|----------------|------|
| 「とりあえず3」で全部揃える | 必ずクイック判定 → 6軸で評価する |
| 完璧主義による過大見積り | スコープ外は別issueにする |
| テストを考慮しない | テスト軸は必ず評価に含める |
| 不確実性を無視 | 調査タスクはSP 1〜2で別issueにする |
| 見積り時間と実績時間の乖離を振り返らない | 完了後に実績SPを記録し次回見積りに活かす |
| 調査と実装を同一issueで見積もる | 不明点が多い場合は調査issueを先に切り出す |
